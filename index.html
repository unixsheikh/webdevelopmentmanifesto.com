Web Development Manifesto

~ for a more sane web ~

Version: 1.0.0. Last updated: 2022-03-14

Introduction

Complexity is the source of bloated, hard to use, error prone, and inconsistent software. With complex code and complex solutions, problems are solved in suboptimal ways, valuable resources are endlessly tied up or wasted, performance slows to a halt, and vulnerabilities become commonplace.

For the past two or three decades the main solution to complex, bloated and slow software has been to simply throw more hardware at the problem. This is no longer acceptable because it is not sustainable.

Furthermore, software developers need to take responsibility for the work they do. Software development is a craft and making good software requires both skill and patience. Following hype and trends simply cannot do. If you're a software developer you need to take pride in the work you do, you cannot do that if you know that the product you create is slow, bloated and unnecessarily complex.

We used to sit around in the Unix Room saying, 'What can we throw out? Why is there this option?' It's often because there is some deficiency in the basic design — you didn't really hit the right design point. Instead of adding an option, think about what was forcing you to add that option.

― Doug McIlroy

Designing simple and elegant software is far more difficult than letting ad-hoc or over-ambitious features obscure the code over time. However one has to pay this price to achieve reliability and maintainability. Furthermore, minimalism results in reasonable, secure and attainable goals. In software simplicity is a virtue that can neither be overstated in importance, nor substituted by anything else.

The crucial behaviour we call "security" really just means the software performs the intended goal only, and cannot be convinced to perform additional unintended behaviours. If you don't know precisely what you putting into the mixing bowl and why you shouldn't be making a cake, get a job in sales or something and save us all the grief. Or go ahead, use a bunch of draino and some matchheads, but don't share it.

― Theo de Raadt

Simplicity is the heart of the Unix philosophy. The more code lines you have removed, the more progress you have made. As the number of lines of code in your software shrinks, the more skilled you have become and the less your software sucks.

With the "Web Development Manifesto" we strive to maintain minimalism and clarity to drive web development to a completion that result in a much more sane World Wide Web.

Terminology

The difference between a website and a web application

If the primary function of the site is to provide information in the shape of text and images, without any additional user interaction besides clicking on anchor tags and filling out input fields, then the site is a website and not a web application. Such a site should be fully capable of running in any kind of web browser, even a console based browser such as Lynx.

Google Maps or a side-scolling shooting game on the other hand is a web application, they both require on-the-fly update of content and sophisticated user interaction.

Manifesto

  1. We NEVER follow hype and trends blindly

    The IT industry has turned into a fashion industry in which many companies and startups follow what is trending rather than what is most sane and pragmatic. If everyone else is doing it, it is automatically assumed that it must be right. It's just like The Emperor's New Clothes

    Much of the hype originates from Silicon Valley and it has more to do with sales and marketing than it has to do with good software engineering.

    Rather than following hype and trends we analyze the problems that we're trying to solve and investigate which technical solutions are best suited to solve our problems. Not only from a pure business perspective, as in how fast can we get something up and running, but also in high regard for security, performance, environmental concerns and user experience and freedom.

  2. We NEVER make a website depend on JavaScript

    If we use any JavaScript for enhanced user interaction, we always make sure that we have validated that the functionality is actually truly enhancing the experience, and that the website still works 100% in a browser with JavaScript disabled.

  3. We STRIVE to use native JavaScript

    Bulky and bloated JavaScript frameworks and libraries are often to blame for the horrible and slow user experience most websites provide today.

    We strive hard to only make use of native JavaScript or small and lean JavaScript libraries.

  4. We NEVER use programming languages that are not performance optimized and well suited for web development

    Programming languages such as Go and PHP (despite its syntactic irregularities) are both performance optimized and well suited for web development.

    When speed matters even more, C, C++ and Rust are other options.

  5. We NEVER use Django or Ruby on Rails

    As famous as these systems are for quickly making websites, just as slow are they in server performance. Python and Ruby are extremely ill suited for web development. The performance penalty is expensive and scaling is difficult.

  6. We NEVER use frameworks

    Web development in Go or PHP requires no framework as they both contain a web application framework in and of themselves. Putting a framework on top of yet another framework only adds unnecessary complexity.

    The only exception to this rule is if the development is done in C, C++ or Rust as these programming languages don't provide a build-in web application framework. Yet, we still strive to avoid complex frameworks and prefer small and efficient libraries.

  7. We STRIVE to avoid object oriented programming

    When people use the object oriented paradigm they tend to focus much more on paradigm details and design patterns rather than solving programming problems. Classes become dependent on classes that depend on yet even more classes in order to adhere to so-called rules of correct object oriented design just to write "hello world" on the screen.

    You can put your functions into classes for organization purposes, that's fine, but other than that the object oriented paradigm adds little of true value. We strive to avoid it.

  8. If we use PHP, we NEVER use the single entry point controller pattern

    The web server is the single entry point, it is the router by its very nature. We do not add yet another router on top of that only to rewrite every single request on the server before it gets served, that defeats the very architecture we're building on. Both rewrites and redirects has a performance cost.

  9. We NEVER use ORMs

    An ORM library is a library written in your programming language of choice that encapsulates the code needed to manipulate data in a database, so you don't use SQL anymore. Instead you interact directly with an object in the same programming language you're using.

    ORMs add unnecessary complexity and result in a lower performance. We only talk to the database in the native language of the database, such as SQL.

  10. We STRIVE hard to know and understand the software we use

    A supply chain attack in software relies on our ignorance of the software we use or our inattentiveness to malicious changes. We must always strive to know and understand as much as possible about the code we use.

    Any time we pull in code we haven't verified and understood, such as module dependencies or libraries, we're dealing with a possible supply chain attack.

  11. We ALWAYS follow the security guidelines from OWASP

    The OWASP Top 10 is a standard awareness document for developers and web application security. It represents a broad consensus about the most critical security risks to web applications. The OWASP Web Security Testing Guide (WSTG) Project produces the premier cybersecurity testing resource for web application developers and security professionals.
